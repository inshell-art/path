# Audit Module Draft for `opsec-ops-lanes-template`

## 1) Intent
Add a first-class **ops audit module** to the template so downstream repos can:
- run lanes for execution control (`bundle -> verify -> approve -> apply -> postconditions`)
- run audits for independent assurance (process correctness, policy compliance, traceability)

This module is for **operational/process auditability**, not smart-contract security analysis.

## 2) Scope and Non-Goals
### In scope
- Audit artifacts and schemas
- Audit scripts and runbook in scaffold
- Control catalog mapped to existing lane artifacts
- Agent response contract for audit requests
- CI entrypoint for periodic audit checks

### Out of scope
- Formal external security audit workflow
- Vendor-specific compliance frameworks (SOC2/ISO-only language)
- Legal attestations

## 3) Relationship to Lanes
- Lanes produce evidence as part of execution.
- Audit consumes that evidence and checks whether controls were actually followed.
- Lanes are per-change; audits are periodic or release-gated.

## 4) Naming and Repo Identity
### Recommendation
Keep repository name as `opsec-ops-lanes-template` for now. Add audit as a module.

Reason:
- Existing downstream references and subtree/submodule paths stay stable.
- No migration churn while adding significant new behavior.

## 6) Data Contracts
### `audit_plan.json` (`schemas/audit_plan.schema.json`)
Minimum fields:
- `audit_id` (string)
- `network` (`devnet|sepolia|mainnet`)
- `scope` (object)
- `time_window` (object with `from`, `to`)
- `run_ids` (array of lane run ids)
- `controls` (array of control ids)
- `auditor` (string)
- `repo_commit` (string)

### `audit_evidence_index.json` (generated by collect)
Minimum fields:
- `audit_id`
- `bundle_refs` (array)
- `files_indexed` (array of `{path, sha256}`)
- `collect_time`

### `audit_report.json` (`schemas/audit_report.schema.json`)
Minimum fields:
- `audit_id`
- `status` (`pass|pass_with_findings|fail`)
- `coverage` (controls covered / total)
- `summary` (counts by severity)
- `verified_claims` (array)
- `limitations` (array)
- `repo_commit`
- `generated_at`

### `findings.json` (`schemas/audit_finding.schema.json`)
Per finding:
- `id`
- `severity` (`low|medium|high|critical`)
- `control_id`
- `title`
- `evidence_refs` (paths/hashes)
- `repro_commands` (array)
- `status` (`open|accepted|resolved`)
- `owner`

### `signoff.json`
Minimum fields:
- `audit_id`
- `approver`
- `approved_at`
- `report_hash`
- `notes`

## 7) Control Catalog (v1)
Define control IDs in `docs/audit-controls-catalog.md`.

Recommended v1 controls:
1. `AUD-001` Bundle integrity hash verified.
2. `AUD-002` `run.json` commit pin matches expected code state.
3. `AUD-003` Lane policy file exists and lane id is valid.
4. `AUD-004` Approval hash matches bundle hash.
5. `AUD-005` Apply used approved bundle only (no ad-hoc args).
6. `AUD-006` Postconditions present and status recorded.
7. `AUD-007` Rehearsal proof gate respected when required.
8. `AUD-008` Signer allowlist + lane signer mapping respected.
9. `AUD-009` Network/chain consistency across artifacts.
10. `AUD-010` Secrets hygiene evidence recorded (no leaks in diff/log bundle).

## 8) Script Behavior (Scaffold)
Add deterministic scripts under `examples/scaffold/ops/tools/`.

### `audit_plan.sh`
- input: `NETWORK`, `AUDIT_ID`, optional `RUN_IDS`
- output: `audits/<network>/<audit_id>/audit_plan.json`

### `audit_collect.sh`
- reads plan
- indexes bundle files from `bundles/<network>/<run_id>/`
- writes `audit_evidence_index.json`

### `audit_verify.sh`
- executes control checks against indexed artifacts
- writes intermediate verification results
- non-zero exit on control execution errors

### `audit_report.sh`
- composes `audit_report.json` and `findings.json`
- enforces schema validation

### `audit_signoff.sh`
- writes `signoff.json`
- ties signoff to `sha256(audit_report.json)`

## 10) Agent Contract Extension
Update template agent docs and snippet:
- Add an **Audit Response Contract** for prompts like:
  - "run audit"
  - "show audit findings"
  - "what did audit verify"
- Keep same structure as ops response contract:
  - `Minimal Evidence Pack`
  - `Common Answer`
- Additional audit requirement:
  - findings-first ordering (critical -> low)
  - explicit distinction between `VERIFIED` and `INFERRED`

## 11) Policy Additions
Add `policy/audit.policy.example.json` with:
- `required_controls` by lane/network
- `min_coverage_percent`
- `max_open_findings_by_severity`
- `audit_frequency` (`per_release|weekly|monthly`)
- `required_signoff_roles` (names only; no secrets)

Downstream can copy to:
- `ops/policy/audit.policy.json`

## 12) CI/CD Hooks (Template Guidance)
Add example workflow notes:
1. Scheduled `audit-all` for non-prod.
2. Pre-release gate:
   - fails if `high/critical` open findings above threshold.
3. Upload audit artifacts as immutable build artifacts.

## 13) Downstream Adoption Sequence
1. Pull template update.
2. Copy audit policy example into local `ops/policy/`.
3. Add audit scripts/Make targets.
4. Run first `devnet` audit over recent run ids.
5. Tune thresholds and control strictness.
6. Enable release gate on `sepolia/mainnet` branches.
